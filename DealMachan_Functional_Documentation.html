Platform Foundation

System Overview

Deal Machan is a deal, coupon, and loyalty platform that connects customers, merchants, and administrators within a structured multi-sided marketplace. The platform enables merchants to publish and manage discount offers, loyalty programs, and promotional campaigns, while customers discover offers, redeem coupons, and earn rewards. Administrators oversee all platform activity, approve content, manage users, and generate operational reports.

The platform operates through four distinct applications that work together via a shared database, with the API acting as the communication bridge between front-end applications and data storage. The Admin application has its own direct database access path for operational management functions.

Data Architecture

Module Relationships

Merchant → Store

A Merchant must have at least one Store to offer coupons, flash discounts, and store coupons. Every coupon, store coupon, and flash discount can be associated with a specific store within the merchant's portfolio.

Store → Category

Stores are classified using a many-to-many pivot relationship with categories and sub-categories via store_categories. This drives store discovery by category in the Customer Application.

Coupon → Merchant / Store

Every coupon belongs to a Merchant and optionally to a specific Store. Coupon searches and home-feed filtering leverage both the merchant's location (via stores) and coupon content.

Coupon → Redemption → Customer

When a coupon is redeemed, a redemption record links the coupon, customer, store, merchant, discount applied, and transaction value. This record drives all merchant analytics.

Coupon → Subscription (Customer Saves)

Customers save coupons to their wallet through a coupon_subscriptions record, separate from redemption. This allows merchants and admins to track demand before redemption occurs.

Store Coupon → Gift → Customer

Merchant creates store coupon → assigns to customer → customer sees in wallet → redeems at store. A fully directed gift chain distinct from the public coupon flow.

Analytics → Coupon Redemptions

Merchant analytics are computed entirely from the coupon_redemptions table. Filtering by merchant_id across redemptions provides all KPI data.

Subscription Plan → Feature Gate

The merchant's subscription_plan value is embedded in the JWT on login. Every protected API endpoint checks the plan against the Feature Gate before processing the request.

Home Feed → Five Modules

The customer home screen assembles data from: Advertisements, Flash Discounts, featured Merchants (by coupon count), top Coupons (by save count), and featured Tags from master data — all in one API response.

Coupon Approval → Notification System

When an admin approves or rejects a merchant's coupon, the Notification System delivers an in-app notification to the merchant informing them of the decision.

Contact Enquiries → Lead Management

Business sign-up enquiries are the entry point for the merchant acquisition pipeline. Sales admins can follow up and create acquisition leads from enquiry details.

Gift Coupons → Coupon Wallet

The Gift Coupon module depends on Coupon Management for the source coupon. On acceptance, the gifted coupon enters the customer's wallet and becomes redeemable through the normal flow.

Card Configuration → Card

A card configuration defines the plan a customer selects. When selected, the configuration's id, validity period, and expiry date are written to the customer's cards record, tying the physical card to a plan tier.

Active Card → Coupon & Contest Actions

Saving, subscribing, and redeeming coupons, plus contest participation, all require the customer to have an active card. The API enforces this via the requireActiveCard middleware before processing any of these requests.

Data Model

Entity Relationships

Core Entities
User
The foundational identity record for all platform participants. Stores login credentials (email, phone, password) and user type. Both customer and merchant profiles are linked to a user record through a one-to-one relationship.
Merchant
Represents a registered business linked to one user account. Has multiple stores, a subscription plan, and status. Associated with labels, tags, and a priority weight affecting search position.
Customer
Represents a registered consumer with type classification (standard, premium, deal maker), subscription status, referral code, city assignment, and optional loyalty card linkage. The linked card may carry a configuration tier (silver, gold, platinum, diamond) that gates platform actions such as coupon saving and redemption.
Store
A physical location owned by a Merchant with city/area assignment, operating hours, contact info, and gallery images. Store-category assignments managed through store_categories pivot.
Coupon
A promotional offer linked to a Merchant and optionally a Store. Has discount type, value, validity window, usage limit, approval status, and banner image. Linked to categories via coupon_categories, cities via coupon_city_targets, and tags via coupon_tags.
Store Coupon
A merchant-issued coupon assigned directly to a specific customer. Targeted and gifted rather than publicly browsable. Tracks assignment and redemption state per recipient.
Flash Discount
A time-sensitive store-level percentage discount that auto-expires once its end datetime passes. Independently browsable on the customer home screen.
Engagement Entities
Review
Links a customer to a merchant with rating, text, and merchant reply. Auto-approved on submission.
Grievance
Links a customer to a merchant (optionally referencing a coupon). Has open/resolved status. The grievance_messages table stores the threaded correspondence.
Message / Notification
Powers in-app direct messaging between merchants and customers, and the admin notification system. Notification type classification controls display styling.
Contest / Survey / Referral
All engagement modules link participation records back to individual customers, allowing customers to track their own activity within each program.
Loyalty & Program Entities
Card
Loyalty card with card number, variant, status (unassigned, activated, blocked), preprinted flag, customer assignment, and an optional link to a card_configurations record. When linked, expiry date, days remaining, and expiry status (active / expiring_soon / expired) are computed and stored.
Card Configuration
Admin-defined card plan: name, classification tier (silver / gold / platinum / diamond), price, validity in days, HTML features list, max live coupon slots, coupon authorisation tinyint, card image front/back, and optional city restrictions via card_config_cities. Publicly selectable configurations are surfaced via the /public/card-configurations API endpoint.
Card Config Partners
Ordered list of partner merchants associated with a card configuration. Each partner has a type (premium / normal), optional image, optional URL, and sort order. Returned as a nested array with the configuration detail.
Card Auth Codes
Time-limited 6-digit codes generated for pre-printed card activation. Each code is consumed on successful use.
Dealmaker Application / Tasks
Application records and individual tasks assigned to approved Deal Makers, with checklist, progress status, and reward tracking.
Mystery Shopping Tasks
Admin-assigned evaluation tasks for selected customer-shoppers targeting specific merchants, with JSON checklists and payment tracking.
Commercial & Administrative Entities
Coupon Redemption
Transaction record linking coupon, customer, store, merchant, discount amount, and transaction value. The primary data source for all analytics.
Sales Registry
Manual transaction entries logged by merchants to record sales not tied to a specific coupon redemption.
Coupon Bulk Allotment / Items
Parent record groups the batch; individual allotment items record per-recipient assignments for pre-allocated coupons.
Audit Log
Immutable record of every significant admin action with actor, action type, affected entity ID and table, timestamp, and IP address.
Advertisement / Blog Post / CMS Page
Promotional banners, editorial blog content, and static informational pages — all managed through the Admin Application with scheduling and slug routing.
Structural Entities
City / Area / Location
Hierarchical geographic reference data. Stores and customers are associated with cities and areas for location-aware content filtering.
Category / Sub-Category
Hierarchical classification system for stores and coupons driving discovery by category in the Customer Application.
Tag / Label / Profession / Day Type
Free-form and reference data for merchant tagging, customer profile enrichment, and important day tracking. All managed as configurable master data.

Integration Architecture

Cross-Application Interactions

Customer App ↔ Backend API
Every interaction travels as an authenticated HTTP request with JWT access tokens. Browsing uses public endpoints (no auth required). Saving coupons, reviews, contests, and personal actions call authenticated customer endpoints. The API returns location-aware content when a city ID is provided.
Merchant App ↔ Backend API
All merchant operations — store config, coupon publishing, redemption scanning, analytics — are processed through the Backend API. The merchant JWT carries the subscription plan and store access scope, which the API uses on every request to enforce feature gates and store-level restrictions.
Admin App ↔ Database (Direct)
The Admin Application operates through server-side rendered PHP controllers that interact with the database directly — it does not call the Backend API. Admin actions (approvals, content creation, user management, configuration) are written directly to the database and immediately affect what the API serves.
Admin App → Merchant App (Indirect)
Admin approvals change fields like profile_status directly in the database. On the next merchant login or API request, these changes are reflected. Coupon approvals, subscription changes, and notification broadcasts all flow from Admin to end-user applications via the shared database.
Merchant App → Customer App (Indirect)
Merchants publish content (coupons, stores, flash discounts) through the Merchant Application via the Backend API. This content is then served by the same Backend API to customers. The two front-end applications never communicate directly — all interaction is mediated through the API and shared database.
Notifications Flow (Admin → All)
When an Admin broadcasts a notification, the Admin Application bulk-inserts notification records into the notifications table for each target recipient. The Backend API surfaces unread notifications to customers and merchants when they next check their notifications feed.
