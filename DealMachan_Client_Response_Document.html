1 Confirmed Bugs & Errors — Critical Fixes

These are verified code-level defects found during the codebase analysis. They require immediate attention and will be fixed in the next release.

⚠ BUG Error #1 — Review Rating Submission Fails
Client reported: "Error is review submission." / "Review rating submission."

Root Cause Identified. There are two distinct defects causing this:

  • Backend (API bug): In api/controllers/Public/StoreBrowseController.php → submitReview(), the authenticated user's user-account ID ($auth['id']) is being stored directly as the customer_id in the reviews table. However, reviews.customer_id is a foreign key that references the customers table (which has its own auto-incremented primary key separate from users.id). This causes either a foreign-key constraint violation (review is rejected silently) or stores an incorrect ID, breaking the review linkage permanently. The same wrong ID is also used in the duplicate-review check, meaning a customer can submit the same review multiple times or get blocked when they shouldn't be.
  • Frontend (silent error): In customer/src/pages/stores/StoreDetailPage.tsx, the reviewMutation has only an onSuccess handler and no onError handler. As a result, when the backend rejects the submission, no error message is shown to the customer — the form appears to do nothing, which is confusing.

Fix Plan: (a) Resolve the correct customer.id from the authenticated user.id before inserting the review, matching the pattern used in all other customer controllers. (b) Add an onError callback to the frontend mutation to display a meaningful error toast.

⚠ BUG Error #2 — Add Coupon (Admin): Error + All Entered Data Is Lost
Client reported: "Add coupon from admin account ERROR – Data is also getting flushed."

Root Cause Identified. When the admin coupon creation form fails validation (e.g., missing coupon code, discount value of zero, or a duplicate code), the controller performs a server-side redirect back to the blank /coupons/add page. Because it is a redirect (a new HTTP GET request), all form data typed by the admin is lost. This is a standard redirect-after-POST issue not handled gracefully in the current implementation.

  • Additionally, if a database exception occurs during the INSERT, there is no catch block in the create path, which may surface a white-screen PHP error rather than a styled error message.

Fix Plan: (a) Store all POSTed form data in the PHP session before redirecting on validation failure, and repopulate the form fields from session data on the next GET request (standard flash-input pattern). (b) Wrap the database INSERT in a try/catch block with a clear, user-friendly error message on failure.

● ENHANCE Error #3 — Add Coupon: Categories Must Filter by Merchant's Registry
Client requested: "Add coupon – First select a Merchant & Store so the category and sub category is filtered to their registry – will select from this list. / explain category targeting / location targeting."

Current Behavior: The Add Coupon form in the admin panel currently loads the entire list of all platform categories globally, regardless of which merchant is selected. The store dropdown only appears when a merchant is selected on the Edit page (not the Add page).

What Will Be Changed:

  • The Merchant dropdown will be the first mandatory selection. Once a merchant is chosen, the Store dropdown will dynamically populate with only that merchant's registered stores (via an AJAX call). This already works on the Edit page and will be ported to the Add page.
  • After selecting a merchant and/or store, the Category and Sub-category selector will reload to show only the categories registered under that merchant in the merchant_categories table. This prevents mismatched targeting.

Category Targeting explained: Categories on a coupon define the type of offer (e.g., Food & Dining, Electronics). Customers browsing by category will find the coupon. Sub-categories provide finer filtering (e.g., Coffee Shops under Food & Dining).

Location Targeting explained: A coupon can be restricted by city (the "City Targets" section) or further narrowed to specific areas or named locations within a city. Only customers whose selected city/area matches will see the coupon in their feed.

● PARTIAL Registration with Pre-Assigned Card Number & Auth Code
Client requested: "If Self Generated with a Card number → signup → enter card number → Auth Code → Select City → Select DOB → I am a "

Current State: The card activation with auth code API is implemented in api/controllers/Customer/CardController.php. The auth code is a 6-digit time-limited code (expires in 30 minutes) that is sent as a notification to the card issuer (merchant/admin) who relays it to the customer. Security Note: The current auth code generation uses random_int(0, 999999) padded to 6 digits — this is cryptographically secure. The auth code is verified as an OTP (time-limited, single-use) and sent only to the issuing entity's notification inbox, not directly to the customer. This matches the client's stated requirement of verifying the customer's mobile via the issuer.

● CLARIFICATION Card Upgrade — Card Number Retained; Expiry = 1 Year from Join Date; Rate Rules
Client stated: "If a customer is upgrading to a new card, card number remains the same. Expiry to Subscription is one year from the date of joining – if not its disabled automatically. No rate applies if the card is lifetime."

Understood and Noted. Implementation Plan:

  • When a customer upgrades a card, only the card_configuration_id (and thus the tier/variant) will be changed on the existing card record; the card_number will not change.
  • Expiry will be set to 1 year from the original joining (activation) date for paid/standard subscriptions. For Lifetime subscriptions, the expiry_date column will be NULL, and the system will never auto-disable such cards.
  • The nightly auto-disable job (or a cron that checks expiry_date < NOW()) will mark expired cards as inactive.
  • When an admin renews or upgrades a customer's card, the system will not allow a price to be applied (the rate field will be disabled/hidden in that admin action per client's instruction).

✓ CONFIRMED G1 — No Delete Permission for Now

Already Enforced. The codebase does not expose delete actions for customers, merchants, or cards through the standard admin UI. The only deletions available are for configuration-level items (such as deleting an unused card configuration) and are protected by a "linked records" check. This policy will be maintained throughout all new feature additions.

● CLARIFICATION G2 — Merchant Registration: Label & Priority / Premium Partner Logic
Client asked: "While registering a merchant there is an option Label & Priority / premium partner – how is the logic?"

Current Behavior Explained:

  • Label: A merchant can be assigned a display label (e.g., "Featured", "New", "Verified") from the Labels master list. This label appears as a badge on the merchant's listing card in the customer app. It is purely cosmetic/marketing — it does not restrict any functionality.
  • Priority Weight: An integer value (e.g., 0–100) stored as priority_weight on the merchant record. When merchants are listed in search results or on the home page, those with higher priority weights are sorted first. This allows giving certain merchants greater visibility without affecting other settings.
  • Premium Partner flag (is_premium): A boolean that marks a merchant as a "Premium Partner." Premium Partners receive a distinct visual treatment in listings (e.g., a gold badge, featured placement). In the public store browse API, premium merchants are ordered before non-premium ones alongside priority weight.

Summary: Label = display badge; Priority Weight = sort order in listings; Premium Partner = featured tier with special UI treatment. All three are independent and can be used in combination.

Client requested: "Super admin to have a separate passcode for Review Tab. From here a store's rating can be modified + or – in percentage. Add a review & Rate in false account. (NAME) (EMAIL: MASKED)."

Current State: The admin review section only allows approving, flagging, or restoring customer-submitted reviews. No rating override or synthetic review capability exists.

Plan:

  • Separate Passcode: A special passcode (secondary authentication, in addition to the main admin login) will be required to access the protected review management actions. This will be stored as a hashed secondary PIN on the super admin's record and verified before sensitive actions.
  • Rating Modifier: An internal adjustment mechanism will allow the super admin to apply a positive or negative percentage modifier to a store's displayed average rating (e.g., "Adjust displayed rating by −5%"). The raw reviews remain unchanged; only the displayed aggregate is adjusted. The modifier is stored transparently in the system log.
  • Synthetic Review Entry: A form to add a manually authored review under a fictional display name. The NAME will be displayed; the EMAIL will be stored internally but masked on the public-facing display (e.g., j***@gmail.com). This capability will be access-controlled exclusively by the super admin passcode.

Important Note: Synthetic reviews and rating modifications are powerful tools that affect customer trust. We strongly recommend that these actions are always recorded in the Audit Log with the super admin's identity, timestamp, and reason (an optional notes field). This protects the platform from legal or reputational risk by maintaining an internal record of changes.

● NEW Loyalty Card Mandatory for All Customer Activities (Except Reviews & Grievances)
Client stated: "Loyalty Card – mandatory for all customer activities excluding review rating and grievances (should prompt to acquire a card from the list)."

Current State: Customer activities (coupon saving, contests, surveys, referrals, etc.) do not currently enforce loyalty card ownership.

Plan: A loyalty card guard will be added in the customer app. Whenever a logged-in customer (who has no active loyalty card) attempts to perform any activity other than submitting a review or a grievance, they will see a prompt: "You need a DealMachan Card to do this. Get your card now!" with a button linking to the card selection wizard. This will be handled at the API level as well (a 403 response with code CARD_REQUIRED) and at the frontend routing/action level.
